# TCS-OCR取り込み判断（販売OCRシステム）リライト版

販売OCRの機能をTCS-OCRにどう取り込むかをまとめた判断資料です。
TCS-OCRの既存実装を調査し、その結果と突き合わせて 3 分類で整理しました。
仕様メモは `販売OCRシステム.md` を参照してください。

---

## 分類軸

| 分類 | 意味 |
| --- | --- |
| **拡張** | TCS-OCRの既存機能を土台に、一部追加すれば実現できる |
| **新規** | TCS-OCRに該当機能がなく、新しく作る必要がある |
| **確認** | 仕様が曖昧で、実装判断の前に要件調査が必要 |

完全に既存機能でカバーできる項目は、本資料から除外しています。
除外した項目は末尾「判断から外した項目」にまとめました。

### 分類サマリ

- **拡張（7件）**
  - 1. 業務OK / NG判定
  - 2. TIFF（OCR入力画像）の業務識別子リネーム保存
  - 3. バウンディングボックスの「業務エラー赤枠」表示
  - 4. マスタ参照UI（ポップアップ）
  - 5. 修正履歴ログ出力（監査トレイル）
  - 6. エラー時のページ遷移ガード
  - 7. 画像ビューアの拡大・縮小・ドラッグ移動
- **新規（5件）**
  - 8. 業務エラーチェックロジック群（ID存在 / 店番Mod10 / 「?」混入 / 入力不備）
  - 9. マスタDB / API（店舗・商品台帳）
  - 10. 汎用データ作成（基幹システム向け変換）
  - 11. 外部システムへのデータ転送（FTP / SFTP / REST API）
  - 12. データ保持期間管理 / 自動削除
- **確認（2件）**
  - 13. 電子帳簿保存法対応
  - 14. 注文用紙テンプレート定義の移行スコープ

---

## 拡張

### 1. 業務OK / NG判定

概要: OCR結果を基幹システムへ連携してよいか（OK / NG）を判定する機能。既存のレビュー状態は「人による確認の有無」のみを管理しており、業務的な可否を表す軸を追加する。

元仕様: 4-2 OK/NGデータの分離

OK / NGの条件:
「業務的に送ってよい（OK）」とは、業務エラーが一つも無いこと。業務エラーは販売OCR仕様（4-3-1）の4つのチェックで定義され、これらを全て満たせばOK、一つでも外れればNGとなる。
- IDが台帳に実在すること（存在しないIDはNG）
- 店番のモジュラス10検算が合うこと（合わなければNG）
- 認識結果に「?」が残っていないこと（残っていればNG）
- 入力に不備がないこと（数量があるのに品番が空ならNG）

ただし「?」のチェックはTCS-OCRでは見直しが要る。TCS-OCRは読めない箇所を「?」ではなく信頼度で表すため、「?」の有無ではなく信頼度による判定へ置き換える（詳細は新規 8 の「TCS-OCRに合わせた要否の再評価」）。

前提として、OCRジョブが成功していること、および店番・品番などの値が取り出せること（テンプレートに名前付き領域が定義されていること。IDチェックには台帳が必要）が要る。

要確認: 4つのチェックに加えて人によるレビュー完了も送信の条件に含めるか、各チェックを送信ブロックとするか警告にとどめるか、一部だけNGのとき注文全体を止めるか。いずれも業務側の確定が必要。

既存基盤:
- レビューステータス（未確認 / 完了 / 修正完了）は導入済み
- ただしこれは「人が確認したか」を表すもの

追加すべきこと:
- 「業務NG（基幹システムに連携できない状態）」を表す状態の追加
- 業務エラーチェックの結果から、自動でNG状態へ遷移させる仕組み（チェック本体は「新規」側）

**🚨！！コメント  
　既存OCRの「NG」という概念は登録してある帳票以外のレイアウトのものがOCRされたとの意です。  
　→複数ファイルの帳票をOCRする場合、途中でNGファイルがあっても後続のOCR処理は続けます。   
！！コメント**  


設計時の判断点（コード確認の結果）:
既存のレビューステータスは「人が確認したか」の軸で、「業務上連携してよいか」とは別の軸です。コードを確認したところ、このサマリー状態は、人のレビュー操作のたびにブロック単位の確認状況から再計算される派生値でした。そのため同じ項目に業務NGの値を足すと、次のレビュー操作で上書きされて消えてしまいます。したがって業務NGは、既存の状態への値追加ではなく、専用の項目（別フィールド）として持つべきです。なおサマリー状態は固定の列挙型ではなく長さ20の文字列のため、専用項目を追加するコストは小さく済みます（列を1つ足すだけで対応できます）。

### 2. TIFF（OCR入力画像）の業務識別子リネーム保存

概要: 入力画像を、業務識別子（日付・注文用紙ID・店番）を含む名称で保存する機能。現状はジョブ単位のフォルダに固定名で保存しており、識別子は付与されない。

元仕様: 4-2 / 4-3 TIFF画像のリネーム保存（`年月日_注文用シートID_店番_OCR認識後.tiff`）

既存基盤:
- ジョブ単位のフォルダに入力ファイルを保存する仕組みは実装済み
- 保存形式は入力の種類で分かれる
  - jpg / jpeg / bmp / webp / gif → PNGに変換して保存
  - png → そのまま保存
  - それ以外（TIFFやPDFを含む）→ 変換せず元データのまま保存
- 販売OCRの入口であるTIFFは、現状は元データのまま固定名で保存され、業務識別子は付かない

追加すべきこと:
- 業務識別子（注文用紙ID / 店番 / 日付）を用いた名称での保存、またはリンク作成
- 命名規約を設定として保持する仕組み

仕組み上の注意点:
- TIFFを変換せず元データのまま保存するのは意図的。変換処理は1枚の画像を前提にしており、TIFFは1ファイルに複数ページを持てるため、変換するとページが壊れる恐れがある。それを避けて元のまま保管している。
- リネームに使う店番・シートIDは、ファイル保存の時点には存在しない。これらはOCRで画像を読み取って初めて分かる値で、しかもOCRが「ここが店番」と判別できるのは、テンプレートに名前付きの領域が定義されているからである。つまりリネームはOCR結果に依存し、その前提として確認 14（テンプレートの名前付き領域）が要る。
- リネーム名「年月日_店番_シートID」は1ファイル＝注文用紙1枚を前提にしている。FAX1通に複数枚が1つのTIFFとして届く場合、1ファイルに店番・シートIDが複数組入り、この命名規約が破綻する。複数枚の扱いは業務に確認が要る。

そもそもリネームが必要かの見直し:
旧システムがファイル名に業務識別子を埋め込んだのは、ファイルを探す手段がファイル名しかなかったため。TCS-OCRにはデータベースと検索画面があり、入力ファイルの情報（元の名前・投入日時・保存パス）はデータベースに記録され、結果画面で日付や状態によって絞り込める。つまり「業務識別子で探せるようにする」という目的は、TCS-OCRでは既にデータベースと検索で果たされている。
したがってリネームの正当性は、内部で探すためではなく、外向きの用途に絞られる。
- 電帳法対応（確認 13）。電子保存したファイルを書面や外部へ出すとき、人が見て分かる名前が要件になりうる。
- 外部システムへの連携（新規 11）。相手が業務識別子付きのファイル名を要求する場合。
内部で検索したいだけなら、ファイル名を書き換えるより、データベースに業務識別子を持たせて検索可能にする方がTCS-OCRの流儀に合い、電帳法の検索性要件とも一致する。リネーム機能を作る前に、用途が内部検索か外部要件かを確認する。
なお、データベースと検索画面が在るのは事実だが、「だからリネーム不要」「DB索引化が良い」は設計判断であり、用途の確定は業務確認による。また現状の絞り込みは画面側のフィルタで、大量データには別途サーバ側の索引化が要る。

！！コメント**  

設計時の判断点:
TIFFのまま保持するか、PDF/A等へ変換するかは「確認 13: 電帳法対応」と合わせて別途決めます。

### 3. バウンディングボックスの「業務エラー赤枠」表示

概要: 業務エラー（ID不在・チェック違反など）の該当箇所を、画像上に強調表示する機能。既存の色分けはOCRの信頼度に基づくもので、業務エラーとは別の指標である。

元仕様: 4-3 / 4-3-1 エラー箇所の赤枠表示・赤文字表示

既存基盤:
- 認識結果の枠を、OCRの信頼度（高 / 中 / 低）で色分け表示する仕組みは実装済み
- 信頼度が低い箇所は、既に赤枠で表示される

追加すべきこと:
- 業務エラー（ID不在 / 店番チェック違反 / 入力不備）を保持するデータ（エラー種別・対象項目・位置）

**🚨！！コメント　
  業務エラー（ID不在 / 店番チェック違反）を仮で入れておけばいいです。  
 　顧客の要望やマスターがどこにあるかなどを聞かないとわからない内容です。  
   現時点では、仮DBを作ってそこを参照する程度で良いと考えます・  
！！コメント**  
  
- 信頼度の表示とは別のチャネルで、業務エラーを示す（色ではなくアイコン。詳細は下記の表示方針）
- 業務エラーのある行のみを絞り込む表示

コード確認と表示方針:
- 信頼度の色分けは枠線だけでなく背景の塗りも使っており、低信頼は既に赤（枠線・背景とも）。業務エラーも赤にすると低信頼と見分けがつかず、かつ1つの枠は同じ場所に1色しか持てないため、信頼度と業務エラーを重ねて表示できない。
- そこで業務エラーは色ではなく別チャネルで重ねる。該当箇所に警告アイコンを重ね、エラー理由は枠内に詰めず右パネルや一覧側に表示する（業務エラー行の絞り込みと統合）。背景を赤で塗る案は低信頼の塗りと衝突するため使わない。
- 表示すべき業務エラーは旧仕様より範囲が狭い。「読めない」箇所は既に低信頼の赤枠で示されるため、拡張3で新たに示すのはID不在 / 店番Mod10 / 入力不備の3つ。「?」混入は信頼度で扱う（新規 8 の要否の再評価を参照）。
- 業務エラーの位置（アイコンや枠の座標）は項目（テンプレート領域）単位で、領域の定義（座標を持つ）から取る。OCR結果そのものは領域IDまでしか持たない。

**🚨！！コメント  
  別添資料で  
！！コメント**  

設計時の判断点:
枠の描画基盤は流用できます。ただし「OCRの信頼度の低さ」と「業務上の不正」は意味が異なるため、データとしては分離して保持する必要があります。

### 4. マスタ参照UI（ポップアップ）

概要: 確認画面上で、店番・品番をマスタ台帳と照合するためのポップアップ。類似のポップアップは存在するが、辞書登録用で用途が異なる。

元仕様: 4-3 マスタ参照（ID / 店番 / 品番の検索UI）

既存基盤:
- 選択テキストを受け取って辞書に登録するポップアップが既にある
- ただしこれは「登録」用であり、「検索・照合」用ではない
- 流用できるのはポップアップの構造（選択文字を渡す導線、対象選択のドロップダウン、入力欄）

追加すべきこと:
- 「マスタ参照ポップアップ」を新規に作成（既存ポップアップの構造を参考にする）
- 動作は登録ではなく「マスタに存在するか確認 / 値を取得して表示」

役割の見直し:
旧システムでこの参照UIが確認の主役だったのは、人が手で台帳を引いて実在を確かめるしか手段がなかったため。TCS-OCRでは、ID存在チェックを自動で行う仕組み（新規 8）と、業務エラーをアイコンで知らせる表示（拡張 3）を作れば、それらが先に動く。これらはまだ実装されておらず作ることが前提だが、それを作れば「正しいか確かめる」主役は自動チェック側に移り、手動の参照UIは補助に変わる。仕様自身もマスタ参照を「確認用途のみ」と位置づけており、検証の本筋ではない。
- 残る用途は「確認」ではなく「修正」。業務エラーが出た項目を直すとき、正しい値を台帳から検索して見つけるための道具として使う。
- さらに、台帳を照合した結果（例: 店番に対応する店名）を項目の脇やエラー理由の欄にそのまま表示すれば、人がポップアップを開く手間も減らせる。ポップアップ前提にしないインライン表示も検討できる。
- 位置づけと優先度: 拡張4は検証の主役ではなく修正の補助。自動チェック（新規 8）と台帳本体（新規 9）が先に要るため、優先度はそれらより低い。
以上は新規 8 を作る前提での設計判断であり、検証の主役を自動側に置く方針は業務と要相談。

**🚨！！コメント　
  マスター参照は顧客がどこにマスタDBを持っているかにもよります。現時点では腹にDBを抱えればよいかと思います。  
  既顧客では、会計ソフト（SQLServer）にアクセスしたり、腹にDB抱えたりしています。
  腹に抱える場合はマスタメンテを作ってます。
！！コメント**  


設計時の判断点:
照合先のマスタ（台帳）本体は「新規 9」で別途必要です。画面部品としては新規ですが、既存ポップアップの構造・配置・操作を踏襲できるため「拡張」として扱います。

### 5. 修正履歴ログ出力（監査トレイル）

概要: 訂正の操作履歴（担当者・日時・内容）を記録し、外部出力できるようにする監査用機能。現状は「日時・訂正後の内容」までしか保持していない。

元仕様: 4-3 電帳法対応の修正履歴ログ出力（CSV）

既存基盤:
- 訂正操作の履歴を残す「追記専用」の仕組みは既にある（アプリは追記のみで書き換え・削除をしない運用。ただし強制の有無は後述）
- 1操作ごとに「状態・訂正後のテキスト・操作日時」が記録される

追加すべきこと:
- 「担当者」の記録 ― 現状は保持していない
- 「訂正前のテキスト」の記録 ― 現状は訂正後のみ保持
- CSV / JSON での履歴出力

TCS-OCRの特性と本当の壁:
この追記専用の仕組みは、システム全体の流儀（物理削除せず追記で残すIDM）と同じ。旧システムなら監査ログを後付けする必要があるが、TCS-OCRは最初からデータを追記で残す作りで、修正履歴もその形。土台は馴染んでいる。
ただし電帳法を満たすには、埋まっていない壁が三つある。
- 担当者（誰が）: ログイン利用者を識別する認証がそもそも無い（クライアントは匿名前提）。修正者を記録する元情報が無く、認証の導入が前提になる。最大の壁。
- 改ざん防止: 「追記専用」はアプリの運用でそうしているだけで、データベース側の強制（書き換え・削除の禁止）は無い。法的にはDBレベルの制御やタイムスタンプ・ハッシュチェーン等が要る（確認 13）。
- 原本画像の保全: 監査で「この訂正は元のどの画像に対するものか」を辿るには原本画像が残っている必要がある。だが入力画像は全ジョブ共有のフォルダに保存され並行投入で上書きされうる既知の不具合があり、原本に遡れない恐れがある。拡張5の前提として先に直す必要がある。

設計時の判断点:
「日時・訂正後の内容」は取得できますが、「担当者・訂正前の内容」は現状残りません。監査要件次第で記録項目の追加が必要です。担当者の記録にはログイン利用者を識別する仕組み（認証連携）が前提となるため、その状況次第で着手前に「確認」へ回す可能性があります。

### 6. エラー時のページ遷移ガード

概要: エラーが解消するまで後続画面への遷移を抑止する制御。既存の遷移制御は結果画面への「進入」を対象としており、必要なのは結果画面からの「離脱」抑止で、方向が逆である。

元仕様: 4-3-1 エラー時のページ移動禁止

既存基盤:
- 「OCRが成功するまで結果画面に進入できない」制御は実装済み（処理中・失敗・未着手はすべて進入不可）

追加すべきこと:
- 「結果画面 → 後続画面（基幹送信・完了など）」への遷移可否を判定する仕組み（販売OCRが求めるのはこの方向）
- 遷移不可の条件: レビュー未完了 / 業務エラーが残存 / NG状態
- 抑止時のメッセージ表示

**🚨！！コメント　
  マスターDBとの照合の部分は、正常になるまで「確認完了」を活性にしない。その他の項目はあくまで運用の目検になるので
  「確認完了」をOKとする。
！！コメント**  


目的に立ち返った再構成（設計提案）:
以下はコードで確認できる事実ではなく、4-3-1の目的を解釈した設計提案である。
旧仕様の「ページ移動禁止」は、画面が一直線の作りだった旧システムの手段であって、本当の目的は「エラーやNGのあるデータを基幹へ送らせない」ことだと解釈できる。TCS-OCRの結果画面は一覧と詳細を行き来する閲覧型のため、全離脱をロックするとユーザーを画面に閉じ込めることになり相性が悪い。
そこで、移動そのものを止めるのではなく、危険な副作用である送信アクションを止める方が素直になる。
- 送信ボタンの実行可否を「業務エラー無し かつ NGでない」で判定する。これは拡張 1 のNG状態を流用でき、既存の実行可否パターンと同じ流儀。送信先の画面・機能は新規 11 で作る。
- ただし画面側の実行可否だけでは不十分。利用者は画面制御を回避できるため、基幹送信のAPI側でも同じ業務エラー / NG判定を行う二重化が要る。
- 離脱時はハードロックではなく、エラーを残して離れる際の軽い警告にとどめる案も検討できる。
この見方では、拡張6は独立した遷移ガード機構というより、拡張 1（NG状態）と新規 11（送信）に概ね統合される。「全離脱ロック」か「送信ガード」かは業務と確認する。
なお既存の進入ガードは、OCR実行直後の「結果を表示」ボタンに限定で、メニュー経由の進入は無条件である。

設計時の判断点:
既存は「結果画面への進入可否」、必要なのは「結果画面からの離脱可否」で、方向が逆です。流用できるのは判定制御の実装パターンに留まります。

### 8. 業務エラーチェックロジック群

概要: 連携前の妥当性検証。ID存在・店番の検算（Mod10）・「?」混入・入力不備を自動判定する。現状この種の検証は実装されていない。

元仕様: 4-3-1 エラーチェック項目

内訳:
- ID存在チェック（マスタ照合）
- 店番のチェック計算（Mod10方式による検算）
- 「?」文字チェック（OCRが認識できなかった箇所の検出）
- 入力不備チェック（数量はあるが品番が無い等）

既存状況:
- TCS-OCRの辞書は「誤読補正」用であり、マスタ照合や値の検証はできない
- 値の妥当性を検証する仕組みは現状無い

TCS-OCRに合わせた要否の再評価:
4-3-1の4チェックは旧・販売OCRの特性に合わせたもので、そのままTCS-OCRへ移植すべきではない。特に「?」チェックは見直しが要る。
- 「?」混入: 旧システムは読めなかった文字を「?」で出力していたため、このチェックが必要だった。TCS-OCRのエンジンは「?」を未認識の印として出さず、「読めない」を信頼度（読み取りの確からしさを表す数値）で表す。そのため「?」チェックはほぼ空振りになる。等価なのは「信頼度が一定未満ならNG」という判定で、信頼度が低い箇所の赤枠表示は既にある（拡張 3）。ただしLLMエンジン（Gemini / Qwen）は信頼度を常に最大値で返すため、信頼度による未読検出が効かない点に注意。
- ID存在 / 入力不備: OCRエンジンに依らない業務上の検証なので、引き続き必要。
- 店番Mod10: 店番のコード体系が現役なら有効。信頼度では拾えない「自信ありげな誤読」を補える独立した価値がある。ただし刷新候補ではコード体系そのものの見直し論もあるため、現役かどうかを業務に確認する。
- 結論: 「?」チェックは信頼度による判定へ置き換える。ID存在と入力不備はそのまま。店番Mod10は業務確認のうえ判断する。

材料は揃っている（実装の軽さ）:
チェックの対象となる値（店番・品番・数量・ID）は、OCR結果に名前付きの項目として既に取り出せる形で入っている。よって4種のチェックは「入力を取り出すところから作る」のではなく「判定の算法を足す」に近く、思ったより軽い。
- 店番Mod10 / 入力不備 / 読めない検出は、入力が揃うため外部依存なし。
- ID存在チェックだけ照合先の台帳（新規 9）に依存する。
- いずれもテンプレートに「店番」「品番」等の名前付き領域が定義されていること（確認 14）が前提。
ただし算法・業務エラーを保持するデータ・台帳照合のいずれも未実装で、「外部依存はID存在のみ」は実装確認ではなく仕様からの推論である。

必要なもの:
- 4種類のチェック処理
- チェック結果を業務エラーとして保持するデータ（「拡張 3」と連携）

元仕様: 4-3 マスタ参照の参照先（Accessファイル / CSV）

既存状況:
- 該当するマスタは存在しない（店舗・商品・取引先のマスタが無い）

方式の選択肢（自前で持つと決め打ちしない）:
旧システムがマスタを手元のファイルで持っていたのは、マスタへ問い合わせる手段が無かったため。本来、店番・品番の正本は基幹システム側にあるはずなので、TCS-OCRが自前で台帳を抱える以外の道もある。

必要なもの（案A＝自前で持つ場合）:
- 店舗マスタ / 商品マスタのデータ定義と取込フロー
- 参照用API

依存関係:
マスタ参照UI（拡張 4）と業務エラーチェック（新規 8）の双方が、このマスタに依存します。

### 10. 汎用データ作成（基幹システム向け変換）

概要: OCR結果を基幹システムが取り込める形式へ変換する機能。現状は自システム用の形式のみ出力する。販売OCRの中核機能。

元仕様: 4-3 汎用データ作成

既存状況:
- OCR結果は出力されるが、基幹システム向け形式への変換は無い

実態と前提:

必要なもの:
- 変換ルールの定義（テンプレートごと）
- 変換処理（レコード組み立て・項目マッピング・出力）

位置づけ:
販売OCRの中核機能です。これがないとOCR結果を基幹システムへ連携できません。

### 12. データ保持期間管理 / 自動削除

概要: 保存年数を設定し、期限切れデータを自動削除する機能。現状、保存期間の管理も削除手段も持たない。

元仕様: 4-3 保存期間・自動削除（999 で無制限）

既存状況:

- ジョブ結果は手動でも削除できない

**🚨！！コメント　
  既システムはそもそもジョブ結果は管理してないと思う  
！！コメント**  

必要なもの:
- 保存期間の設定（全体 / テンプレートごと / ジョブごと。粒度は要設計）
- 期限切れデータを削除する定期処理

削除をめぐるTCS-OCRの特性:
このシステムはデータを物理削除しない流儀（IDM）で、マスタの「削除」も中身は論理的なアーカイブ（後から戻せる）。ただし全体が物理削除ゼロというわけではなく、関連テーブルのDB削除やテンプレート画像のファイル削除は実在する。削除の実態は層で分かれる。
- マスタ（辞書など）: アーカイブによる論理削除があり、戻せる。
- ジョブの記録: アーカイブの仕組みすら無く、現状そもそも消せない。
- 入力・結果のファイル: 削除する手段が無く溜まり続ける。消すなら物理削除で、戻せない。
「自動削除」と一括りにすると、戻せるDBの論理削除と戻せないファイルの物理削除が混ざって危険。二つを分けて設計する。不可逆の危険は主にファイル側にある。

段階実装と前提:
- 削除は取り消せず、保存年数は電帳法（7〜10年）に依存する（確認 13）。法令が固まる前に実削除を作って動かすのは危険。
- 期限の設定と、期限切れ候補を一覧表示するだけの空実行（dry-run）までは法令確定前でも安全に先行できる。実削除（特にファイル）は確認 13 の確定後につなぐ。
- 定期実行の足場として、メール監視の定期処理が参考になる（汎用のスケジューラは無い）。

依存関係:
保存年数は電帳法（7〜10年）に依存するため、「確認 13」と合わせて設計します。

### 13. 電子帳簿保存法対応

概要: 取引書類の電子保存に関する法令（電子帳簿保存法）への対応。改ざん防止・検索性・保存期間が要件となるが、まず必須要件の特定が先行する。

電子帳簿保存法（略して電帳法）は、国税関係の帳簿・書類（注文書・請求書・領収書など）を電子データで保存する際のルールを定めた法律です。
紙の代わりに電子保存する場合、次の 3 点が求められます。

- 後から改ざんできないこと
- 定められた期間、確実に保存すること
- 必要なときに検索・出力できること

**🚨！！コメント  
✅ FAX原本画像  
✅ OCR結果データ  
✅ 修正履歴  
✅ 画像とデータの紐付け情報  
!!コメント**  

販売OCRは注文書（取引書類）を扱うため、読み取った画像・データはこの法律の対象になりうるものです。

主な要件は次の 3 つです。

- 真実性の確保 ― 改ざん防止。タイムスタンプの付与、または訂正・削除履歴の保持
- 可視性の確保 ― 取引年月日・取引金額・取引先での検索、ディスプレイ・書面への速やかな出力
- 保存期間 ― 原則 7 年（欠損金がある事業年度等は最長 10 年）

販売OCR仕様には「10 年保存」「TIFFリネーム」「親機SQLite登録」の 3 点しか記載がありません。
これだけでは上記の法令要件を満たすか判定できないため、Phase 1 着手前に要件を確定する必要があります。

TCS-OCRには現状、電帳法対応として設計された機能はありません。
関連しうる部品も別目的で実装されたものです。要件別の対応状況は次のとおりです。

| 電帳法の要件 | TCS-OCRの現状 | 判定 |
| --- | --- | --- |
| 真実性（改ざん防止） | 時刻の記録はあるが、アプリ内の処理時刻に限られる。法的に有効な時刻認証（タイムスタンプ局）や電子署名は無い | 未対応 |
| 真実性（訂正・削除履歴） | 訂正履歴は追記専用で保持される（アプリが書き換え・削除をしない運用。DBレベルの強制は無く、直接操作すれば改変は可能）。「担当者」「訂正前のテキスト」も保持しない | 部分的 |
| 可視性（検索性） | 一覧はファイル名・日付・ステータスで絞り込める（画面側のフィルタ）。「取引金額」「取引先」での検索はできない。値そのものは項目ごとに構造化されているが、DBの列として索引化されていないため検索に使えない（索引化すれば届く位置にある） | ほぼ未対応 |
| 可視性（画面・書面出力） | 画面表示は可能。帳票としての出力・印刷は無い | 部分的 |
| 保存期間（7〜10 年） | 保存期間管理・自動削除のいずれも無し（項目 12）。期限を考慮した保存もできない | 未対応 |

関連するが目的が異なる部品:
- 入力ファイルの照合用ハッシュはあるが、用途は重複検知であり改ざん検知ではない
- ファイル保存はあるが、TIFFは変換・改名されず元データのまま保存される（命名規約・長期保存の設計なし）
- さらに入力画像は全ジョブ共有のフォルダに固定名で保存され、並行投入で上書きされうる既知の不具合がある（別記、Codex確定）。原本画像が確実に保存される保証が現状ないため、画像保存・真実性の前提を直接崩す。電帳法対応の前に修正が要る
- データベースはPostgreSQL。親機のSQLiteより堅牢ではあるが、それ自体が電帳法対応を意味するものではない

確認すべきこと:
- 対象範囲 ― どのデータ / どの画像が電帳法の対象か
- 保存形式 ― PDF/A 等の指定の有無
- 検索性 ― 日付 / 取引先 / 金額等での検索可否
- 訂正・削除履歴 ― 既存のレビュー状態 + 修正履歴ログ（拡張 5）で要件を満たすか
- タイムスタンプ / 改ざん防止 ― 必要な方式（電子署名 / ハッシュチェーン等）
- 保存先 ― ローカルストレージ / クラウドストレージ（S3 / Azure Blob）の選定

要件が確定すれば、以降は「新規」へ移します。

### 14. 注文用紙テンプレート定義の移行スコープ

概要: 注文用紙のレイアウト定義（各項目の印字位置）をTCS-OCRに用意する。格納先のテンプレート機能はあるが、既存データを移行するか新規作成するかが未確定。

4-1（注文用紙作成・印刷）は取り込まない方針です。
ただし、注文用紙のレイアウト情報（注文用紙ID / 店番 / 品番が紙のどこに印字されるか）はOCR側で必要になります。

TCS-OCRのテンプレート機能（領域を座標で名前付きで登録できる）に格納すれば、技術的には対応可能です。
格納能力は確認済み。初期化・シナリオ投入用に、テンプレートと領域をまとめて投入するスクリプトが在り、これを移行に流用できる可能性があります。

実は土台であること:
確認14は分類上「確認」の一つだが、実装上は他の複数項目が乗る土台。注文用紙のテンプレートに「店番」「品番」等の名前付き領域と座標が定義されて初めて、次が動く。
- 拡張2（リネームに使う店番・シートIDの値）
- 拡張3（業務エラーの位置 ＝ 領域の座標）
- 新規8（チェックの入力 ＝ 領域の値）
- 新規10（基幹レコードへのマッピング ＝ 領域の値）
分類は「確認」だが、実装の順では最初に固めるべき。

確認すべきこと:
- 既存の販売OCRのレイアウト定義（Access上の構成データ）をテンプレートへ移行するか、TCS-OCR上で新規作成するか
- 移行する場合の変換プログラムの要否
- 注文用紙の種類（版の違い・店舗ごとの違い）をどの単位で管理するか

## 取り込まない

| 機能 | 理由 |
| --- | --- |
| 4-1 注文用紙作成・プレ印刷 | OCRの前工程にあたる別業務。OCRの動作には不要で、実装コストが大きく、OCR本体への影響もない。事業上の必要性が確定してから別フェーズで検討する（注文用紙のレイアウト定義移行のみ「確認 14」で扱う） |
| OCR後の自動印刷 | 紙運用の要否は顧客次第。標準機能とする必要はなく、運用が固まってから追加すればよい |
| 二重FAX対応 | 元の販売OCRでも未対応と明記。FAX機器側で運用回避すべき問題であり、OCR側で対処するものではない |

## 判断から外した項目（既存機能で完全にカバー）

以下は TCS-OCR の既存実装で完全にカバーされているため、本資料の判断対象から除外しました。
判断の妥当性を検証する際の参考として記録します。

| 販売OCRの機能 | TCS-OCR既存実装 |
| --- | --- |
| OCR認識処理本体 | OCRエンジン管理機能（Azure / Google / PaddleOCR / YomiToku / Gemini / Qwen VL 等に対応） |
| OCRエンジン選択 | ジョブ作成時にエンジンを指定できる |
| テンプレート選択 | ジョブ作成時にテンプレートを指定できる |
| 辞書（共通 / 領域別）による誤読補正 | 辞書機能。領域ごとの割り当てに対応 |
| 画像補正（回転 / 傾き / ノイズ除去） | OCRエンジン側で実施（TCS-OCRからはエンジンに委譲） |
| 中間データの保存 | 入力・生レスポンス・結果・エラーをジョブごとに保存（入力は種類により変換 or 元のまま。詳細は拡張 2） |
| マウス・キーボード操作両対応 | 全画面でキー操作に対応、マウス操作も標準対応 |
| データ修正画面のDB反映（店番修正等） | レビュー状態の更新で同等の処理を実現 |
| フォルダ監視（受信トリガ） | フォルダ監視機能（パターン / 優先度 / 重複検知） |
| メール監視（受信トリガ） | メール監視機能（メール巡回 / 送信者パターン / 重複検知） |
| 親機SQLite | TCS-OCRは当初からPostgreSQL（移行不要） |
| マスタファイル（Access） | AccessはWSL/Linuxで扱えないため、TCS-OCRでは「マスタDB / API化」が前提（新規 9 参照） |

再検証で判明した訂正: 上の表のうち以下は「完全にカバー」ではなく、要再評価。
- 辞書（領域別）: 共通辞書は効くが、領域別の割り当ては実際のOCR処理に繋がっておらず効かない（設定する場所はあるが配線されていない）。
- 画像補正: 常に掛かるのではなく、オプションを指定したときだけ。
- マウス・キーボード操作: 主要な3画面のみ。辞書・監視・テンプレート管理の画面はキー操作に未対応。
- データ修正画面のDB反映: 確認画面の修正は編集テキストの履歴として残るだけで、OCR結果や店番・業務のデータは更新しない。販売OCRの「店番修正→業務DBへ反映」とは別物で、同等扱いは撤回する。
- メール監視: 本体だけではジョブを作らない。受信からOCR実行までは、別プロセスのメール監視アプリ（OcrViewer.Mail）が起動していて初めて成立する。
特に「修正が業務データに反映されない」点は、拡張 1・新規 8・新規 10（修正した値を基幹へ流す）の前提に直結するため、対面で共有する。
